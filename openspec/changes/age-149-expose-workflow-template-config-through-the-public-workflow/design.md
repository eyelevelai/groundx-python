## Goals / Non-Goals

**Goals:**
- Confirm the correct mechanism for delivering `WorkflowTemplate` and the `template` field on `WorkflowRequest` / `WorkflowDetail`: Fern regeneration from the updated `eyelevel-fern-config` OpenAPI spec, not hand-editing.
- Verify that the `.fernignore`-fenced hand-written layer is uninvolved and untouched.
- Establish which hand-written file (README.md) requires a targeted edit, and what that edit covers.
- Confirm that `WorkflowRequest` and `WorkflowDetail` are tolerant readers and that the additive touchpoint poses no compatibility risk.
- Gate the CI verification path: mypy + pytest must pass post-regen.

**Non-Goals:**
- Hand-editing any Fern-generated file (types, client, raw_client, `reference.md`).
- Modifying `src/groundx/ingest.py`, `src/groundx/csv_splitter.py`, or `src/groundx/extract/`.
- Implementing `customSteps`, `outputRoutes`, `leafFields`, or any other custom workflow-step scope.
- Altering the `.fernignore` list (no new paths need protection — `openspec/` is already listed; README.md is already listed).

## Decisions

### D1: Delivery mechanism — Fern regen, not hand-editing

**Decision:** The `WorkflowTemplate` type and the `template` field on `WorkflowRequest` / `WorkflowDetail` are delivered exclusively by running Fern regen (`fern generate`) against the updated `eyelevel-fern-config` OpenAPI spec. No generated file is edited by hand.

**Rationale:** `src/groundx/types/workflow_request.py`, `src/groundx/types/workflow_detail.py`, `src/groundx/types/__init__.py`, and `src/groundx/workflows/client.py` each carry `# This file was auto-generated by Fern from our API Definition.` as their first non-blank line. Editing them directly is futile — the next regen overwrites every change. The upstream spec (`eyelevel-fern-config`) is already FINALIZED (contract.md, Level 1 producer confirmed, commit 1280636); the regen step brings the Python SDK into alignment.

**Alternatives rejected:**
- *Hand-patching generated types* — overwritten on next regen; creates a divergence between the repo state and what `fern generate` would produce. Rejected.
- *Adding `template` via `.fernignore` carve-out + hand-written override* — unnecessary given the spec is already updated upstream; creates a maintained surface with no benefit. Rejected.

**Verification:** After regen, confirm `src/groundx/types/workflow_template.py` exists (or `WorkflowTemplate = typing.Dict[str, str]` appears in an appropriate types file), `template: typing.Optional[WorkflowTemplate]` appears on both `WorkflowRequest` and `WorkflowDetail`, and `WorkflowTemplate` is re-exported from `src/groundx/types/__init__.py`.

### D2: `.fernignore` scope — no changes required

**Decision:** No new paths are added to `.fernignore`. `openspec/` is already listed (survives regen, protecting this change directory). `README.md` is already listed (hand-written, survives regen).

**Rationale:** The change only touches generated types (delivered via regen) and `README.md` (already fenced). There is no new hand-written module, no new test directory, and no new config file requiring protection.

### D3: README.md — targeted hand-written edit

**Decision:** Add a workflow `template` usage example to `README.md` (the only hand-edited deliverable in this change).

**Rationale:** `README.md` is on `.fernignore` and is explicitly hand-authored. The specs require at least one `client.workflows.create(template={...})` example with a `dict[str, str]` value, and a note that omitting `template` is valid. This is a documentation-only edit — no code path is altered.

**Scope of edit:** Add a short "Workflow Template" subsection (or extend the existing workflows section) with a create example and an update example showing `template={"key": "value"}`. Confirm the note that the field is optional and that omitting it leaves the field unset or unchanged.

### D4: Tolerant-reader confirmation (CONSUMER compatibility)

**Decision:** No compatibility intervention needed. `WorkflowRequest` and `WorkflowDetail` already configure Pydantic with `extra="allow"` (both Pydantic v1 via `Extra.allow` and Pydantic v2 via `ConfigDict(extra="allow")`). This is the standard Fern-generated pattern for all types in this SDK.

**Rationale:** The upstream touchpoint is classified Additive in the FINALIZED contract (contract.md). The `extra="allow"` configuration means the current SDK already accepts responses with fields it does not model — including a `template` field in a pre-regen response — without raising a deserialization error. After regen, `template` becomes an explicitly modeled optional field; no change to the Pydantic config is needed. The spec scenario "WorkflowDetail with unknown additive fields still deserializes (tolerant reader)" is satisfied by the existing model config.

**Re-classification check:** The producer contract is Additive — no existing field is removed or renamed, both new fields are optional, no required field is added. The current SDK has no `template` field on `WorkflowRequest` or `WorkflowDetail`, so there is no existing shape being changed. Classification remains Additive; no version/flag/migration mechanism is needed.

### D5: No ADR warranted

**Decision:** No ADR file is written for this change.

**Rationale:** The design decisions here are mechanical applications of the repo's established codegen model — regen-via-Fern is the documented contribution path for API-surface changes, `.fernignore` is stable, and Pydantic `extra="allow"` is the existing SDK-wide pattern. No new cross-cutting pattern, no new tooling choice, and no architectural trade-off that future contributors need a formal record of.

## Risks / Trade-offs

- **[Risk] Regen output diverges from the spec shape** — the FINALIZED contract specifies `WorkflowTemplate = typing.Dict[str, str]` and `template: Optional[WorkflowTemplate]`. If the Fern generator emits a different shape (e.g., `typing.Any` or a nested object), mypy will surface mismatches in any hand-written test that uses the type. Mitigation: the post-regen verification step (mypy + pytest) is the gate; if the regen output is wrong, the CI gate fails and the issue is visible before merge.

- **[Risk] `openspec/` is not on `.fernignore` at regen time** — if `.fernignore` were missing the `openspec` entry, a regen run would delete this change directory. Mitigation: confirmed `.fernignore` already lists `openspec` (line 10 of `.fernignore`). No action required.

- **[Risk] Fern regen `workflow_dispatch` not yet triggered** — the upstream contract notes that regen is manual (`workflow_dispatch`). If regen is not run before the apply step, the generated files in this repo will not yet contain `WorkflowTemplate`. Mitigation: the tasks step will include an explicit "trigger/confirm Fern regen" task as the first step; the post-regen mypy/pytest run confirms the types are present.

- **[Risk] `tests/custom/` tests for workflow may not exist yet** — the spec scenarios require test coverage for `WorkflowDetail.template` deserialization and `WorkflowsClient.create(template=...)` serialization. The existing `tests/custom/` directory may not have workflow-specific tests. Mitigation: new hand-written tests under `tests/custom/` will be added as part of the tasks step to cover the spec scenarios (these are the only hand-authored tests; the generated `tests/` root is regenerated and is not the target).

## Migration Plan

This change is additive with no migration:
1. Confirm `eyelevel-fern-config` upstream spec is FINALIZED (done — confirmed in contract.md).
2. Trigger Fern regen (`workflow_dispatch` on `groundx-python`) to produce updated generated files.
3. Edit `README.md` to add `template` usage example (hand-written, on `.fernignore`).
4. Add hand-written tests under `tests/custom/` covering the spec scenarios.
5. Run `poetry run mypy .` and `poetry run pytest -rP -n auto .` — both must pass.
6. PR → merge → maintainer tags a minor release.

Rollback: if regen introduces an unexpected regression, revert the regen commit. The additive field means no downstream caller is broken by reversion — callers that already pass `template` via the `extra="allow"` escape hatch will continue to work (they were using a dict key not typed in the model; reverting regen merely returns to that state).

Producer contract shapes confirmed — see `contract.md` for the full response shapes.

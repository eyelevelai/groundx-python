# groundx-python — Contributor Guide

This repository is the **official GroundX Python SDK** (`pip install groundx`). It is
the Python equivalent of [`groundx-typescript`](https://github.com/eyelevelai/groundx-typescript) — both are generated from the same upstream Fern API definition by separate generators.

This file is the canonical contribution guide for agents and humans working in this
repo. Other agent-tool surfaces (Codex, Claude, Cursor, Replit, Gemini/Antigravity)
should **route here** rather than duplicate rules — see [§9](#9-other-agent-surfaces).

---

## 1. Setup, install, build, test

The repo uses **Poetry** (pinned to 1.5.1 in CI). Python 3.10+ supported.

```sh
# Install Poetry (matches CI)
curl -sSL https://install.python-poetry.org | python - -y --version 1.5.1

# Install base dependencies
poetry install

# Install with the aiohttp extra (required for async-client tests)
poetry install --extras aiohttp

# Install with the extract extra (required for any work in src/groundx/extract/)
poetry install --extras extract

# Type check (the CI gate)
poetry run mypy .

# Run tests
poetry run pytest -rP -n auto .

# Run only aiohttp-marked tests
poetry run pytest -rP -n auto -m aiohttp .
```

**Before opening a PR:** run `poetry run mypy .` and `poetry run pytest -rP -n auto .`
locally. Both must pass. CI runs the same commands plus the same suite with `--extras
aiohttp` installed.

**Note:** CI runs mypy without `--extras extract`, so extract-related imports fall back
to `Any` via the `ignore_missing_imports = true` overrides. A contributor running
`poetry run mypy .` locally with `--extras extract` installed may surface typing errors
CI doesn't catch. Run mypy with the extras you've actually touched.

## 2. Generated vs hand-written code

Most of this repo is **generated by Fern** from the upstream GroundX API definition.
The generator version and config are pinned in `.fern/metadata.json` — that file also
declares dependency lists, extras, and mypy overrides that get written into
`pyproject.toml` on regeneration.

The boundary between generated and hand-written code is **`.fernignore`**, not the
directory layout. Read `.fernignore` before editing anything in `src/`. Files and
directories listed there survive regeneration; everything else gets rewritten.
`pyproject.toml` is regenerated — to change deps or mypy config, edit
`.fern/metadata.json` (see [§4](#4-the-extract-submodule) for the pattern).

Current `.fernignore` highlights:

| Path | Why it's hand-written |
|---|---|
| `src/groundx/extract/` | The entire extract submodule (see [§4](#4-the-extract-submodule)). |
| `src/groundx/ingest.py` | Hand-written ingest helper (the `client.ingest()` entry point that wraps the generated upload flow). |
| `src/groundx/csv_splitter.py` | Hand-written CSV-splitting utility used by ingest. |
| `README.md` | User-facing README — preserved across regenerations. |

A file's Fern-generated status is also detectable from its first non-blank line: if it
reads `# This file was auto-generated by Fern from our API Definition.`, it is generated
and editing it directly is futile. But `.fernignore` is authoritative — use it.

## 3. Contributing to generated code

Generated code changes happen upstream, not here:

- **API-shape changes** (new endpoints, fields, types) — change the upstream Fern API
  definition. The SDK is regenerated from it and re-released.
- **Generator behavior changes** (how every generated Python file looks — client class,
  error types, serialization, retry policy) — submit a PR to the
  [Fern repository](https://github.com/fern-api/fern), where the Python SDK generator
  lives alongside the other SDK generators.

For a specific file in the generated tree that genuinely needs hand-written behavior,
add it to `.fernignore` (see [§2](#2-generated-vs-hand-written-code)) and write it by
hand. Use this sparingly — every entry is a maintenance commitment.

Hand-written tests that exercise generated code live under `tests/custom/`. Everything
else in `tests/` is regenerated.

To coordinate an upstream API change: external contributors should open a
[GitHub issue](https://github.com/eyelevelai/groundx-python/issues) describing the
needed shape change; a maintainer routes it to the appropriate Fern workspace.
Internal contributors should use the team's existing project-tracking channel.

## 4. The `extract` submodule

`src/groundx/extract/` is the largest hand-written area — agent definitions, prompt
templates, classes, celery tasks, post-processing, services, and settings — the
building blocks for GroundX extraction workflows. End users install it via
`pip install groundx[extract]`.

The extract submodule's public API is **provided as-is and may change between minor
versions.** External code that depends on it should pin a minor-version range.

It has a much larger optional-dep footprint than the core SDK — see
`[tool.poetry.extras]` in `pyproject.toml` for the full list.

**Adding a new optional dep**: edit `.fern/metadata.json`, **not** `pyproject.toml`.
Fern regenerates `pyproject.toml` from the generator config, so direct edits there get
overwritten. The dep needs two entries in `.fern/metadata.json` under
`generatorConfig`:

1. `extra_dependencies.<name>` — declare the dep with `"version": "*"` and
   `"optional": true`.
2. `extras.extract` — append the dep name to the list.

If the dep needs a mypy override (e.g. for missing third-party stubs), also append a
TOML block to `generatorConfig.pyproject_toml` in the same file. Every new dep
enlarges the `groundx[extract]` install footprint — justify it in the PR.

**mypy applies.** `poetry run mypy .` covers extract. Submodule-specific stub gaps are
handled via `[[tool.mypy.overrides]]` blocks declared in
`.fern/metadata.json` → `generatorConfig.pyproject_toml` (the existing overrides for
`groundx.extract.classes.document` and `groundx.extract.tasks.utility` live there).
Editing `pyproject.toml`'s overrides directly will be overwritten on regen.

**No live network in unit tests.** Mock S3, Google Sheets, Redis, OpenAI, etc. Live
integration tests follow the policy in [§5](#5-credentials-and-live-tests).

## 5. Credentials and live tests

CI tests **do not use live GroundX credentials.** Everything under `tests/utils/` and
`tests/custom/` runs against mocked HTTP. A new contributor can clone the repo and run
the full CI test suite without a GroundX account.

If a contribution needs to verify against the live GroundX API:

- **Never commit credentials.** Read `os.environ["GROUNDX_API_KEY"]` in
  contributor-local scripts; gate live tests behind an env var so they skip when the
  key is absent.
- **Do not add a live-tests CI job that requires shared secrets** without coordinating
  with maintainers — there is currently no live-integration job in `ci.yml`.

## 6. Release flow

Releases are triggered by tag pushes. `ci.yml`'s `publish` job runs on any push to a
`refs/tags/*` ref and publishes to PyPI using `$PYPI_USERNAME` / `$PYPI_PASSWORD`
GitHub secrets.

```sh
git tag v3.5.3
git push origin v3.5.3
```

Most commits in this repo are `Release X.Y.Z` markers driven by SDK regeneration.

## 7. Workflow

1. Branch from `main` with a descriptive name.
2. Make changes scoped to the right surface ([§3](#3-contributing-to-generated-code)
   for generated code, [§4](#4-the-extract-submodule) for extract, or other
   `.fernignore`-listed files).
3. Run local gates before pushing:
   ```sh
   poetry run mypy .
   poetry run pytest -rP -n auto .
   ```
4. Open a PR against `main`. CI runs `compile` (mypy) and `test` (pytest, both with and
   without the aiohttp extra). Both must pass.
5. After merge, a maintainer tags a release (see [§6](#6-release-flow)) to ship to PyPI.

**Commit messages.** Clear, descriptive, present tense. Reference the relevant issue
or ticket ID in the subject when applicable.

**Code style.** A `ruff` config lives in `pyproject.toml`. `poetry run ruff check .`
and `poetry run ruff format .` are available, but ruff is not currently a CI gate. Use
ruff on the files *you've changed* (`ruff check <paths>` / `ruff format <paths>`)
rather than the whole tree.

## 8. Package layout

`src/groundx/` contains the SDK. Run `cat .fernignore` for the hand-written paths;
everything else under `src/groundx/` is generated. `reference.md` is also generated
and should not be edited by hand.

## 9. Other agent surfaces

**Do not duplicate this file** into agent-specific variants (`CLAUDE.md`, `CURSOR.md`,
`CODEX.md`, `.cursorrules`, `.replit`, `GEMINI.md`). Duplicated rules drift, then
disagree, then mislead.

Acceptable: a one-line agent-specific file that reads simply `See @AGENTS.md` — the
same pattern the GroundX Studio Harness uses for `CLAUDE.md`. Anything more than that
pointer goes here.

---

## 10. Boundaries

- ✅ **Always do:** run `poetry run mypy .` and `poetry run pytest -rP -n auto .` before pushing, read credentials from `os.environ`, edit code in paths listed in `.fernignore`
- ⚠️ **Ask first:** add a new optional dep to `groundx[extract]`, add a path to `.fernignore`, modify `.fern/metadata.json` or `ci.yml` non-trivially, add a CI job requiring shared GroundX credentials
- 🚫 **Never do:** edit Fern-generated files directly (overwritten on next regen), edit `pyproject.toml` for deps or mypy (use `.fern/metadata.json` per [§4](#4-the-extract-submodule)), remove a path from `.fernignore` without first re-homing the hand-written code, commit credentials in any form, create agent-tool-specific files that duplicate this one (`CLAUDE.md`, `.cursorrules`, etc.)

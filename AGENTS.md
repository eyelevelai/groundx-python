# groundx-python — Contributor Guide

This repository is the **official GroundX Python SDK** (`pip install groundx`). It is
the Python equivalent of [`groundx-typescript`](https://github.com/eyelevelai/groundx-typescript) — both are generated from the same upstream Fern API definition by separate generators.

This file is the canonical contribution guide for agents and humans working in this
repo. Other agent-tool surfaces (Codex, Claude, Cursor, Replit, Gemini/Antigravity)
should **route here** rather than duplicate rules — see [§10](#10-other-agent-surfaces).

---

## 1. Setup, install, build, test

The repo uses **Poetry** (pinned to 1.5.1 in CI). Python 3.10+ supported.

```sh
# Install Poetry (matches CI)
curl -sSL https://install.python-poetry.org | python - -y --version 1.5.1

# Install base dependencies
poetry install

# Install with the aiohttp extra (required for async-client tests)
poetry install --extras aiohttp

# Install with the extract extra (required for any work in src/groundx/extract/)
poetry install --extras extract

# Type check (the CI gate)
poetry run mypy .

# Run tests
poetry run pytest -rP -n auto .

# Run only aiohttp-marked tests
poetry run pytest -rP -n auto -m aiohttp .
```

**Before opening a PR:** run `poetry run mypy .` and `poetry run pytest -rP -n auto .`
locally. Both must pass. CI runs the same commands plus the same suite with `--extras
aiohttp` installed.

**Note:** CI runs mypy without `--extras extract`, so extract-related imports fall back
to `Any` via the `ignore_missing_imports = true` overrides. A contributor running
`poetry run mypy .` locally with `--extras extract` installed may surface typing errors
CI doesn't catch. Run mypy with the extras you've actually touched.

## 2. Generated vs hand-written code

Most of this repo is **generated by Fern** from the upstream GroundX API definition.
The boundary between generated and hand-written code is **`.fernignore`**, not the
directory layout.

**Agent hard stop:** before editing any file in this repo, open `.fernignore` and
confirm the path is listed there or lives under a listed directory. If the path is not
covered by `.fernignore`, do not edit it in this repo. It will be overwritten by the
next Fern generation. This includes generated release/config/runtime files such as
`.fern/metadata.json`, `pyproject.toml`, `poetry.lock`,
`src/groundx/core/client_wrapper.py`, `reference.md`, and most files under
`src/groundx/` outside the hand-written entries below.

If a change seems to require a file that is not covered by `.fernignore`, stop and
route the change to the upstream Fern source or generator instead of patching the
generated output locally:

- **API shape, package metadata, dependencies, extras, versions, or SDK release
  metadata** — change the upstream Fern configuration/API source and regenerate the
  SDK.
- **Generated client behavior or generated headers** — change the Fern generator or
  upstream generator config.
- **A file that should become hand-written** — add it to `.fernignore` first, justify
  the maintenance cost in the PR, then edit it.

Current `.fernignore` highlights:

| Path | Why it's hand-written |
|---|---|
| `src/groundx/extract/` | The entire extract submodule (see [§4](#4-the-extract-submodule)). |
| `src/groundx/ingest.py` | Hand-written ingest helper (the `client.ingest()` entry point that wraps the generated upload flow). |
| `src/groundx/csv_splitter.py` | Hand-written CSV-splitting utility used by ingest. |
| `README.md` | User-facing README — preserved across regenerations. |

A file's Fern-generated status is also detectable from its first non-blank line: if it
reads `# This file was auto-generated by Fern from our API Definition.`, it is generated
and editing it directly is futile. But `.fernignore` is authoritative — use it. A file
without that header is still off-limits unless `.fernignore` protects it.

`reference.md` is also generated and should not be edited by hand.

## 3. Contributing to generated code

Generated code changes happen upstream, not here:

- **API-shape changes** (new endpoints, fields, types) — change the upstream Fern API
  definition, which lives in
  [`eyelevel-fern-config`](https://github.com/eyelevelai/eyelevel-fern-config). That
  repo holds the OpenAPI spec from which this SDK (and
  [`groundx-typescript`](https://github.com/eyelevelai/groundx-typescript)) are
  generated. The SDK is regenerated from it and re-released — do not hand-edit
  generated code here to achieve an API-shape change.
- **Generator behavior changes** (how every generated Python file looks — client class,
  error types, serialization, retry policy) — submit a PR to the
  [Fern repository](https://github.com/fern-api/fern), where the Python SDK generator
  lives alongside the other SDK generators.

For a specific file in the generated tree that genuinely needs hand-written behavior,
add it to `.fernignore` (see [§2](#2-generated-vs-hand-written-code)) and write it by
hand. Use this sparingly — every entry is a maintenance commitment.

Hand-written tests that exercise generated code live under `tests/custom/`. Everything
else in `tests/` is regenerated.

To coordinate an upstream API change: external contributors should open a
[GitHub issue](https://github.com/eyelevelai/groundx-python/issues) describing the
needed shape change; a maintainer routes it to the
[`eyelevel-fern-config`](https://github.com/eyelevelai/eyelevel-fern-config) repo.
Internal contributors should track the change in Linear.

## 4. The `extract` submodule

`src/groundx/extract/` is the largest hand-written area — agent definitions, prompt
templates, classes, celery tasks, post-processing, services, and settings — the
building blocks for GroundX extraction workflows. End users install it via
`pip install groundx[extract]`.

The extract submodule's public API is **provided as-is and may change between minor
versions.** External code that depends on it should pin a minor-version range.

It has a much larger optional-dep footprint than the core SDK. You may inspect
`[tool.poetry.extras]` in `pyproject.toml` to see the generated package shape, but do
not edit that file locally.

**Adding a new optional dep**: do not patch generated package metadata in this repo.
Dependency, extras, version, release metadata, and `pyproject_toml` changes belong in
the upstream Fern configuration/API source, then arrive here through SDK regeneration.
If a dependency is needed only by a hand-written `.fernignore`-protected module,
document the need and coordinate the upstream metadata change instead of editing
`.fern/metadata.json` or `pyproject.toml` locally. Every new dep enlarges the
`groundx[extract]` install footprint, so justify it in the PR.

**mypy applies.** `poetry run mypy .` covers extract. Submodule-specific stub gaps may
need `[[tool.mypy.overrides]]` package metadata. Treat those like dependency changes:
coordinate the upstream Fern metadata/config change and regenerate the SDK. Do not edit
`.fern/metadata.json` or `pyproject.toml` locally.

**No live network in unit tests.** Mock S3, Google Sheets, Redis, OpenAI, etc. Live
integration tests follow the policy in [§6](#6-credentials-and-live-tests).

**Extraction boundary evidence.** For the private four-case extraction certification
chain, `groundx-python` owns the SDK reassembly boundary. Reassembly tests must consume
the extract-stage X-Ray/custom-output packet plus compiled workflow metadata and emit
`workflow_output`, `relationship_output`, `final_output`, diagnostics, and source
provenance for the save/callback boundary. Unit fixtures under `tests/extract/` count
as protected-case evidence only when they consume the previous boundary artifact,
compare actual output to a frozen reviewed expected output, write a machine-readable
diff, preserve hash lineage, and assert the functional output shape. Do not count a
synthetic local fixture as protected-case evidence unless it is the allowed fake
agent/model output emitted by the immediately previous boundary.

Repo-local proof for the SDK reassembly boundary:

```sh
poetry run pytest -q tests/extract/test_custom_output_reassembly.py tests/extract/test_extraction_boundary_reassembly.py
poetry run ruff check src/groundx/extract/custom_outputs.py tests/extract/test_custom_output_reassembly.py
```

**Where extract tests live.** Extract unit tests live under `tests/extract/`, mirroring
the source structure (e.g. `tests/extract/classes/test_group.py` tests
`src/groundx/extract/classes/group.py`). They use `unittest.TestCase` and import extract
code via absolute paths (`from groundx.extract.classes.group import Group`). Run with
`poetry run pytest -rP -n auto`.

Cross-test fixtures shared between extract test files (e.g. sample YAML constants, a
`Source` test-double) live in underscore-prefixed modules like
`tests/extract/prompt/_fixtures.py` — the underscore prefix keeps pytest from collecting
them as test modules. Public test helpers re-exported as part of the
`groundx.extract` API (e.g. `TestField`, `TestXRay`, `TestChunk`, `TestDocumentPage`)
live in `src/groundx/extract/classes/testing.py` and carry `__test__ = False` so pytest
doesn't try to collect them as `unittest.TestCase` classes.

## 5. Extending or adding a hand-written submodule

Two distinct cases for adding hand-written code.

### 5.1 Extending an existing hand-written area (e.g. extract)

Inside `src/groundx/extract/`:

- **New module in an existing subdir** — add the file, update the subdir's
  `__init__.py` to re-export the public surface, optionally update
  `src/groundx/extract/__init__.py` if the symbol should be importable from
  `groundx.extract`. Add a test under `tests/extract/<subdir>/` mirroring the source
  path (see [§4](#4-the-extract-submodule)).
- **New subdir under extract** — create the directory with a hand-written
  `__init__.py` that re-exports the public surface; no `.fernignore` change needed
  (the parent `src/groundx/extract` is already covered).
- **New optional dep** — do not patch generated package metadata in this repo. Route
  dependency and extras changes through the upstream Fern configuration/API source,
  then regenerate the SDK.

### 5.2 Adding a new sibling submodule (like `extract`)

Order matters here:

1. **Add the path to `.fernignore` first.** Otherwise Fern wipes the new directory on
   the next regen.
2. **Create the directory** with a hand-written `__init__.py` that re-exports the
   public surface.
3. **Decide the import surface.** Users will import via
   `from groundx.<submodule> import X` — **not** from `groundx` at the top level. The
   top-level `src/groundx/__init__.py` is Fern-generated and won't know about the new
   submodule's symbols.
4. **Add tests under `tests/<submodule>/`** mirroring the source structure, matching
   the extract convention (see [§4](#4-the-extract-submodule)).

### 5.3 What requires upstream Fern coordination

A new hand-written submodule whose symbols need to appear at top-level
`from groundx import ...` requires an upstream generator-config change. The generated
`.fern/metadata.json` file may show how existing exports arrived, but it is read-only
in this repo. For most new submodules, the `from groundx.<submodule> import X` import
path is the right call.

## 6. Credentials and live tests

CI tests **do not use live GroundX credentials.** Everything under `tests/utils/` and
`tests/custom/` runs against mocked HTTP. A new contributor can clone the repo and run
the full CI test suite without a GroundX account.

If a contribution needs to verify against the live GroundX API:

- **Never commit credentials.** Read `os.environ["GROUNDX_API_KEY"]` in
  contributor-local scripts; gate live tests behind an env var so they skip when the
  key is absent.
- **Do not add a live-tests CI job that requires shared secrets** without coordinating
  with maintainers — there is currently no live-integration job in `ci.yml`.

## 7. Release flow

Releases are triggered by tag pushes. `ci.yml`'s `publish` job runs on any push to a
`refs/tags/*` ref and publishes to PyPI using `$PYPI_USERNAME` / `$PYPI_PASSWORD`
GitHub secrets.

```sh
git tag v3.5.3
git push origin v3.5.3
```

Most commits in this repo are `Release X.Y.Z` markers driven by SDK regeneration.

## 9. Workflow

1. For a non-trivial change, propose an OpenSpec change first (see
   [§8](#8-spec-driven-development-openspec)).
2. Branch from `main` with a descriptive name.
3. Make changes only in `.fernignore`-protected files. Generated-code, package
   metadata, dependency, release/version, or generator-behavior changes go upstream and
   come back through regeneration.
4. Run local gates before pushing — `poetry run mypy .` and
   `poetry run pytest -rP -n auto .`. Both must pass.
5. Open a PR against `main`. CI runs `compile` (mypy) and `test` (pytest, both with and
   without the aiohttp extra). Both must pass.
6. After merge, a maintainer tags a release (see [§7](#7-release-flow)) to ship to PyPI.

**Commit messages.** Clear, descriptive, present tense. Reference the relevant issue
or ticket ID in the subject when applicable.

**Code style.** A `ruff` config lives in `pyproject.toml`. `poetry run ruff check .`
and `poetry run ruff format .` are available, but ruff is not currently a CI gate. Use
ruff on the files *you've changed* (`ruff check <paths>` / `ruff format <paths>`)
rather than the whole tree.

**Comment hygiene.** Use comments for non-obvious behavior, invariants, or risk.
Do not commit PR history, ticket rationale, implementation-plan prose, or
"this test proves..." narration in code or tests; put that context in Linear,
OpenSpec, the PR body, or docs.

## 10. Other agent surfaces

**Do not duplicate this file** into agent-specific variants (`CLAUDE.md`, `CURSOR.md`,
`CODEX.md`, `.cursorrules`, `.replit`, `GEMINI.md`). Duplicated rules drift, then
disagree, then mislead.

Acceptable: a one-line agent-specific file that reads simply `See @AGENTS.md` — the
same pattern the GroundX Studio Harness uses for `CLAUDE.md`. Anything more than that
pointer goes here.

## 11. Boundaries

- ✅ **Always do:** run `poetry run mypy .` and `poetry run pytest -rP -n auto .` before pushing, read credentials from `os.environ`, edit code in paths listed in `.fernignore`
- ⚠️ **Ask first** (open a GitHub issue): add a new optional dep to `groundx[extract]`, add a path to `.fernignore`, change generated package metadata/dependency/release behavior, modify `ci.yml` non-trivially, add a CI job requiring shared GroundX credentials
- 🚫 **Never do:** edit files not covered by `.fernignore`, including `.fern/metadata.json`, `pyproject.toml`, `poetry.lock`, generated client files, generated headers, or `reference.md`; remove a path from `.fernignore` without first re-homing the hand-written code; commit credentials in any form; create agent-tool-specific files that duplicate this one (`CLAUDE.md`, `.cursorrules`, etc.)

## OpenSpec

OpenSpec manages the **documentation lifecycle** for this repo (proposal → specs → design →
tasks). **Implementation** is done with **Superpowers** (brainstorm → plan → TDD → review →
finish), which is ambient in the harness and triggers automatically. All spec work runs inside
this repo on the feature branch.

**Schema:** `spec-driven` (official)   **Role:** `library`   **Profile:** `core`   **Default command:** `/opsx:ff`

Per-artifact content rules live in `openspec/config.yaml`. Inspect templates and runtime
guidance with `openspec instructions <artifact>`.

### Default slash command

`/opsx:ff` — the recommended driver for this repo.
- `/opsx:ff` — small, low-risk changes; all artifacts at once.
- `/opsx:continue` — large or correctness-sensitive flows; one gated artifact at a time.
- `/opsx:explore` — think first; useful when the approach is unclear.

### Skills used by the artifacts

- Open design questions in a proposal → `superpowers:brainstorming`.
- Given/When/Then scenarios in specs → `superpowers:test-driven-development`.
- Architectural decisions → `architectural-decision-records` skill; write ADRs to
  `docs/adr/<LINEAR-TICKET>-<kebab>.md` (Linear ticket prefix mandatory). Cross-service
  decisions live in the **producing** repo and are referenced from consumers' `design.md`.

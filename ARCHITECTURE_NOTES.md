> Lens: **extending SDK features** (hand-written logic layered on the Fern-generated client).
> Repo: `/Users/mayurpabari/workspace/Projects/groundx/groundx-python` · package `groundx` (PyPI) · v3.5.8

## System summary
Official Python client for the GroundX API (RAG / document ingestion + search by EyeLevel).
Most of the library is **auto-generated by Fern** from an upstream API definition (HTTP client,
typed models, per-resource sub-clients). A thin **hand-written** layer adds ergonomics the
generator can't: local-file & directory ingestion (`ingest.py`), CSV/TSV splitting
(`csv_splitter.py`), and a standalone document-extraction toolkit (`extract/`).

## Tech stack
Python 3.10+ · Poetry (pinned 1.5.1 in CI) · `httpx` (sync+async) · `pydantic` · `requests` · `tqdm`.
Optional extras gate heavy features: `aiohttp`, `extract`, `boto3`/`minio`/`gspread`/`openai`/`celery`/`smolagents`.
Quality gates: `mypy 1.13` (strict, the CI compile gate) + `pytest` (asyncio + xdist).

## Entry points
Public `GroundX`/`AsyncGroundX` are re-exported from `ingest.py` (`__init__.py:117`), **not** the
generated `client.py`. They subclass the generated base, so `client.buckets`/`.search`/`.documents`
come from generated code and `client.ingest(...)` is hand-written.

## Core data flow — `client.ingest(local file)` (sync, fast path)
"Ingest local" = **upload-to-object-store-then-ingest-by-URL**. Local bytes never hit the GroundX API.
```
1. GroundX.ingest()            ingest.py:147   hand-written
     prep_documents()          ingest.py:85    split inputs → remote(URL) vs local(path)
2. GroundX._process_local()    ingest.py:484
     split_doc()               ingest.py:134   CSV/TSV → CSVSplitter chunks; else passthrough
     _upload_file()            ingest.py:442
        get_presigned_url()    ingest.py:64    GET upload_api?name=&type= → {URL,Header,Method}
                                               (default host api.eyelevel.ai/upload/file — NOT SDK base_url)
        requests.put(URL, bytes)               direct object-store upload (raw requests, no SDK auth)
        → hosted URL  → rewritten as IngestRemoteDocument(source_url=URL)
3. self.documents.ingest_remote()   documents/client.py:86   GENERATED thin wrapper
4. RawDocumentsClient.ingest_remote() documents/raw_client.py  POST v1/ingest/documents/remote
5. ClientWrapper               core/client_wrapper.py:38   injects X-API-Key + base_url, httpx → network
6. status-code → typed IngestResponse | typed error (BadRequestError 400, UnauthorizedError 401, …)
```
`wait_for_complete=True` adds `_monitor_batch()` (ingest.py:524): polls
`get_processing_status_by_id(process_id)` every 3s until complete/error/cancelled, driving tqdm.
`ingest_directory()` walks dir → batches → funnels back through `self.ingest()`.

## Key abstractions to internalize
1. **Generated-base + hand-written-subclass.** generated = transport + types; hand-written = ergonomics. `.fernignore` = what you may edit.
2. **Two-tier resource clients** (`client.py` typed façade → `raw_client.py` real httpx + status→type mapping). Don't edit raw_client (generated) — change the upstream API def.
3. **Upload-then-ingest-by-reference.** SDK never sends file bytes to the GroundX API; uploads to object store, passes a URL. Every local-file feature becomes a remote-document feature after step 2.

## State & auth
- Client is stateless; only config (api_key) lives on the wrapper.
- Server ingestion is async, keyed by `process_id` in `IngestResponse`; polling is the only progress signal.
- **Two distinct remote services**: upload service (`upload_api`, default `api.eyelevel.ai/upload/file`) vs GroundX API (`base_url` from `environment.py`) — configured independently.
- Auth: `X-API-Key` header on GroundX API calls (client_wrapper.py:38); separate `Authorization: Bearer` branch in async wrapper.

## Seams for change
1. **`GroundX(GroundXBase)` in ingest.py** — add hand-written client methods here (survives regen).
2. **`split_doc`/`_upload_file`/`_process_local`** — the filesystem↔API boundary; new formats/splitters/object-stores plug in here.
3. Format tables `DOCUMENT_TYPE_TO_MIME` / `SUFFIX_ALIASES` (ingest.py:18) — start here to support a new file type.

## Per-module notes

### `ingest.py` (787) — deep-dived
- Helpers: format tables (18–57); batch consts `MAX_BATCH_SIZE=50`, `MAX_BATCH_SIZE_BYTES=50MB`, `override_batch_size`→cap 1000 (59–61); `get_presigned_url` (64, raw requests); `strip_query_params` (76); `prep_documents` (85, URL-check-first); `split_doc` (134, silent `[]` on unsupported).
- Sync `GroundX` (146): `ingest` (147, two modes — wait-path has NO total cap; >max_n guard only on fast path L308); `ingest_directory` (321, rglob+byte/count batching); `_upload_file` (442, PUT-only); `_process_local` (484, filename preserved only when 1 split — L503); `_monitor_batch` (524, 3s poll, 4 near-dup status blocks); `_upload_file_batch` (593, recursively calls self.ingest L629).
- Async `AsyncGroundX` (641): `ingest` simpler — NO batching/wait_for_complete/progress; `_upload_file` (747) is a duplicate and **blocking** (uses `requests.put`, not async).
- Depends on (generated): `self.documents.ingest_remote`, `get_processing_status_by_id`; types Document/IngestRemoteDocument/IngestResponse/IngestStatus; + CSVSplitter.
- Depended on by: `__init__.py:117` only. No other internal callers of the helpers → contained blast radius.

### `csv_splitter.py` (64)
`CSVSplitter(filepath, delimiter)`: splits by `max(rows/1000, file_MB)` (L17). Writes chunks to
`tempfile.mkdtemp()` that is **never cleaned up**. Returns `[original]` if <2 splits.

### `Document` type (types/document.py)
`file_path: str` **required** (no default); model `frozen=True, extra="allow"`. `file_name`/`file_type`/
`filter`/`process_level`/`search_data` optional. Field aliases are camelCase (`bucketId`, `filePath`).
